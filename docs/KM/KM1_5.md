# Kolejne KM dla `library3` — przegląd

## Założenie planu

Ten zestaw KM jest dostosowany do **realnego stanu repo**, a nie tylko do docelowej wizji z dokumentacji.
Przyjmuję więc zasadę:

1. **nie wracamy do foundation DevOps**, jeśli już działa,
2. **najpierw domykamy kontrakt i backend API**,
3. **potem budujemy frontend na stabilnym API**,
4. **na końcu robimy stabilizację MVP i hardening produkcyjny**.

## Co traktuję jako już dowiezione lub częściowo dowiezione

- monorepo `frontend/`, `backend/`, `docs/`,
- root scripts (`lint`, `typecheck`, `test`, `build`, `dev:*`),
- `docker compose` dla local dev,
- workflow `ci`,
- workflow publikacji obrazów do GHCR,
- workflow deployu na Azure VM,
- compose’y produkcyjne i runbook wdrożeniowy,
- podstawowy backend (`/api/health`, szkic `/api/books`, Swagger),
- podstawowy frontend (home + health/books skeleton).

## Najważniejsze luki względem dokumentacji

### API / backend
- brak pełnego modułu `auth`,
- brak modułu `loans`,
- brak modułu `users`,
- `books` są tylko częściowo gotowe,
- kontrakt odpowiedzi i paginacji nie jest jeszcze spójny z dokumentacją,
- nazwy pól domenowych wymagają unifikacji.

### Frontend
- brak tras `/login`, `/register`, `/loans`, `/admin/*`,
- brak kompletnej integracji auth/session,
- brak komponentów MVP (`Navbar`, `LoanModal`, `BookModal`, `Toast/Banner`),
- brak pełnego flow USER i ADMIN.

### Cross-cutting
- trzeba dopiąć testy krytycznych ścieżek,
- trzeba domknąć dokumentację „dla nowej osoby”,
- warto zrobić produkcyjny hardening po dowiezieniu MVP.

## Proponowana sekwencja

1. **KM2** — kontrakt i normalizacja fundamentów domenowych,
2. **KM3** — auth + JWT + RBAC,
3. **KM4** — Books API do poziomu MVP,
4. **KM5** — Loans + Users API,
5. **KM6** — fundamenty FE i integracja auth/API,
6. **KM7** — frontend publiczny + USER,
7. **KM8** — frontend admin + stabilizacja MVP,
8. **KM9** — hardening operacyjny produkcji.

## Dlaczego taka kolejność

- Repo ma już wdrożony tor CI/CD i deploy, więc największa wartość biznesowa jest teraz w **domknięciu funkcji użytkowych**.
- Frontend warto rozwijać dopiero po ustabilizowaniu kontraktu API.
- Milestone operacyjny przenoszę na koniec, bo infrastruktura już istnieje, ale **nie ma jeszcze pełnego MVP**, które trzeba utrzymywać.

## Sposób pracy

Dla każdego KM zakładam:
- osobny milestone w GitHub,
- issue z AC,
- małe PR-y,
- QGP obowiązkowo przed merge,
- smoke test lokalny przez `docker compose`,
- aktualizację README / Swagger, jeśli zmienia się kontrakt.
